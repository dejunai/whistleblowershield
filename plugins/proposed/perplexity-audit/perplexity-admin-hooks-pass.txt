## `admin-hooks.php` — Findings

### 🔴 Critical

**1. `wp_insert_post` hook reads `$_GET['ws_jx_term']` with no nonce or capability check**
The taxonomy auto-assignment hook fires on every `wp_insert_post` for addendum CPTs and blindly trusts the `ws_jx_term` URL parameter . There is no `check_admin_referer()`, no `current_user_can()`, and no verification that the request came from the admin navigation panel. An authenticated user with any post-creation capability (even `contributor`) could craft a URL with an arbitrary `ws_jx_term` value and force a taxonomy assignment on a newly created post. Fix: add `current_user_can( 'edit_posts' )` at minimum, and verify the request is an admin context with `is_admin()`.

**2. `ws_acf_plain_english_guards()` — `$_POST['acf']` walked by field key but field name resolution via `acf_get_field()` can return `false`**
The guard already checks `if ( ! $field_obj ) continue`  — that part is fine. The vulnerability is the `$_POST['acf']` values themselves: `$submitted_plain_english` is set to `trim( (string) $field_value )` directly from `$_POST` without sanitization before the Rule 1 logic runs. The value is never echoed and only used in an `=== ''` comparison, so XSS risk is nil — but if this logic ever evolves to store or display `$submitted_plain_english`, the raw `$_POST` read is a landmine. Downgrading to 🟡 given current usage, but worth noting explicitly.

***

### 🟡 Should Fix

**3. `ws_stamp_verified_by_date()` — reads `$_POST['acf']['field_verification_status']` as a hardcoded field key**
Hook 4 reads the incoming verification status via `$_POST['acf']['field_verification_status']` . This is a hardcoded ACF field key, not a field name. Every other hook in this file reads submitted values by walking `$_POST['acf']` and resolving names via `acf_get_field()`. If the field key ever changes in the ACF config (e.g., during a migration or field group rebuild), this hook silently stops detecting verification transitions — `$incoming` will always be `''`, so the stamp never fires. The same hardcoded key pattern appears in `ws_enforce_source_verify_roles()` (Hook 5) twice . All three should resolve via `acf_get_field()` like the guards do, or the key should be a named constant.

**4. `ws_acf_plain_english_guards()` Rule 3 — deletes `ws_auto_plain_english_reviewed_date` but not `ws_auto_plain_english_reviewed_by`**
The toggle-off cleanup in Rule 3 calls `delete_post_meta` for `ws_auto_plain_english_by`, `ws_auto_plain_english_date`, and `ws_auto_plain_english_reviewed_date`  — but **not** `ws_auto_plain_english_reviewed_by`. The `$_POST['acf']` loop above it zeroes out the ACF field value for `ws_auto_plain_english_reviewed_by`, but that only affects what ACF writes for the field value — the underlying post meta key `ws_auto_plain_english_reviewed_by` is left intact in the database. On the next toggle-on, `ws_acf_stamp_plain_reviewed_by()` checks `if ( get_post_meta( $post_id, 'ws_auto_plain_english_reviewed_by', true ) )` and bails early , so the reviewer is never re-stamped. Add `delete_post_meta( $post_id, 'ws_auto_plain_english_reviewed_by' )` to the Rule 3 cleanup block.

**5. `ws_rebuild_jx_statute_citation_index()` — serialized LIKE query is overly broad**
The meta query uses `'value' => serialize( $statute_id ), 'compare' => 'LIKE'` . `serialize( 5 )` produces `i:5;` — the LIKE search will match any serialized array containing `i:5;`, which includes `i:5;` at any position in any array. So a statute with ID `5` would also match arrays containing `i:50;`, `i:51;`, etc. if MySQL's LIKE isn't anchored. In practice this is low risk because IDs are matched numerically in the first OR branch, and the serialized branch is a fallback — but the LIKE branch could return false positives for IDs that are substrings of larger IDs. The correct pattern uses `serialize( [ $statute_id ] )` or a more precise LIKE value like `%i:' . $statute_id . ';%'`.

### 🔵 Tech Debt

**6. `$ws_stamp_cpts` is a module-level global consumed via `global $ws_stamp_cpts` inside `ws_acf_write_stamp_fields()`**
Same pattern as the `$_ws_dir_*` labels in `render-directory.php` . Works fine solo. Every CPT in the map has the same `author_acf_key` value — the map could be reduced to a flat array of CPT slugs, removing the need for the global entirely.

**7. `ws_citation_save_stash()` and `ws_interp_save_stash()` — dual-use static functions (write + read)**
The stash helpers use a `$ids !== null` branch to toggle between read and write mode . This is a clever pattern but produces confusing call signatures — `ws_citation_save_stash( $post_id )` reads silently, `ws_citation_save_stash( $post_id, [] )` writes an empty array (valid — a citation with no statute). An empty array `[]` is not `null` so the write branch fires correctly. No bug, but the pattern is worth a comment when you revisit it.

***

### ✅ Security Posture — Generally Strong

The field-locking pattern is solid: `'disabled' => 1` means ACF won't process the submitted value on save regardless of DOM manipulation . The role checks on `plain_reviewed`, the source/verify field gates, and the immutability guards on `source_method` and `source_name` are all correctly implemented server-side. The one genuine hole is item #1 — the unauthenticated taxonomy assignment on `wp_insert_post`.
