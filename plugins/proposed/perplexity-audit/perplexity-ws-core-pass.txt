## `ws-core.php`

**Overall: excellent reference document. Two issues.**

### 🟡 `WS_JURISDICTION_TERM_ID` naming convention apology

The inline comment acknowledges this is confusing ("Complaints can be directed to /dev/null") but stops short of fixing it. The constant is used in ~25 places across the codebase as a taxonomy slug. Before launch, a rename to `WS_JURISDICTION_TAXONOMY` (or `WS_JURISDICTION_TAX_SLUG`) would eliminate the permanent cognitive overhead with a single find-and-replace. The comment is a fine interim marker but should have a `@todo pre-launch` tag so it shows up in the audit sweep.

### 🟡 `ws_core_enqueue_assets()` — redundant condition

```php
if ( is_singular() || is_page() ) {
```

`is_singular()` already returns `true` for pages — `is_page()` is a subset of `is_singular()`. The `|| is_page()` branch is dead code. Not a bug, just noise.

### 🟢 `WS_REF_PARENT_TYPES` defined as a bare PHP array constant

PHP 5.6+ supports array constants via `define()`, and WordPress minimum is PHP 7.4+ as of WP 6.3, so this is fine. Worth noting for anyone who sees it and wonders.

### 🟢 Deactivation hook for `ws_feed_monitor_deactivate`

```php
register_deactivation_hook( __FILE__, 'ws_feed_monitor_deactivate' );
```

There's no mention of a feed monitor module in the architecture changelog, and the v3.8.0 feature list doesn't reference it. Looks like a forward stub or a leftover — worth confirming the function is actually defined somewhere in `includes/` and won't trigger a fatal on deactivation.

## `ws-core-front-general.css`

**Overall: clean and well-commented. One issue.**

### 🟡 `.ws-jx-card { display: flex }` but hidden cards set `display: none`

The JS sets `card.style.display = match ? 'flex' : 'none'`. The CSS sets `.ws-jx-card { display: flex }`. These agree. However there's no explicit `.ws-jx-card[hidden]` or `display: none` variant in the CSS, which means if JS is disabled, all cards show (correct). But if the theme or a parent container ever sets `.ws-jx-card { display: block }` for responsive overrides, the JS `'flex'` reset would re-apply flex on show — which would work but could surprise a future developer. Minor, but worth a comment on the `.ws-jx-card` rule noting the JS dependency.

### 🟢 `.copyright-bar { display: none }` — theme override present

The comment correctly documents this is licensed under GeneratePress terms. Fine.

### 🟢 `.ws-tooltip-content` duplicates the `.sr-only` technique inline rather than inheriting it

The `.ws-tooltip-content` rule manually restates all the `sr-only` properties instead of using `@extend` (not available in plain CSS) or just adding the `sr-only` class in JS. Since the JS adds `ws-tooltip-content` as the class (not `sr-only`), the duplicate is necessary. No issue — just noting it's intentional.

## `ws-core-front-jx.css`

**Overall: clean. One gap.**

### 🟡 No styles for `[ws_jx_interpretation]` section

The file header lists the covered shortcodes: `[ws_jx_header]`, `[ws_jx_summary]`, `[ws_jx_statutes]`, `[ws_jx_citation]`, `[ws_jx_limitations]`. But `[ws_jx_interpretation]` was added in v3.8.0 (per `ws-core.php` changelog item #7) and is absent from the CSS header list and has no dedicated ruleset. The interpretation section will render with zero scoped styles — it'll fall through to theme defaults. At minimum the file header should be updated; at most a `.ws-interpretation` block should be stubbed in alongside `.ws-case-law`.

### 🟢 Single responsive breakpoint at 480px

Only the header stacks at 480px. The jurisdiction index card grid has no responsive rules in `ws-core-front-general.css` either (it uses `flex-wrap: wrap` which handles reflow naturally). Fine for now given the pre-launch design-pass caveat.

## `ws-core-front.js`

**Overall: tight and well-structured. One issue.**

### 🟡 Filter tab JS has no keyboard support on buttons

The `.ws-jx-filter-btn` elements are `<button>` elements (presumably — the render layer produces them), so `click` events fire on Enter/Space automatically. But if the render layer ever outputs them as `<div>` or `<span>` with a click listener, keyboard users would be locked out. The JS itself only binds `click` — no `keydown` guard. Add a note in the comment block that this requires the buttons to be native `<button>` elements, or add an explicit `keydown` handler as a safety net.

### 🟡 Version history is stale

The JS version log tops out at `2.4.0`. The file lives inside a `3.8.0` plugin. Any changes made during the 3.x architecture passes (tooltip additions, `sr-only` hint injection) are in the `2.4.0` entry, which predates the whole refactor. The version history in this file should be brought in sync with the plugin version convention — at minimum add a `3.8.0` entry noting it was extracted from `ws-core-front.css` split (since the CSS files both have a `3.8.0 — Split` entry but the JS doesn't).

### 🟢 IIFE pattern, `'use strict'`, `var` throughout

Consistent and appropriate for no-build environments. No issues.

### 🟢 Idempotency guard on external link injection (`.ws-sr-new-tab` check)

Good defensive coding — correctly noted.
