### 2. `acf-jx-statutes.php` — `ws_jx_statute_disclosure_type` uses `save_terms => 0`

Three taxonomy fields in `acf-jx-statutes.php` — `ws_jx_statute_disclosure_type`, `ws_jx_statute_protected_class`, `ws_jx_statute_disclosure_targets`, `ws_jx_statute_process_type`, `ws_jx_statute_adverse_action`, `ws_jx_statute_fee_shifting`, `ws_jx_statute_remedies`, `ws_jx_statute_employer_defense` — all use **`save_terms => 0`**. The data is saved as post meta (arrays of term IDs), not into the taxonomy relationship table.

This is a **deliberate design choice** only if the query layer reads exclusively from post meta. But compare this to `acf-jx-interpretations.php`, where `ws_process_type` uses `save_terms => 1`, and `acf-jx-citations.php`, where `ws_jx_citation_disclosure_type` uses `save_terms => 1`. The inconsistency means you cannot use a single `tax_query` across both statutes and citations for the same taxonomy (e.g. `ws_disclosure_type`) — you'd need a `meta_query` for statutes and a `tax_query` for citations.

**This is almost certainly unintentional.** The `acf-legal-updates.php` changelog from v3.5.0 shows `save_terms` was explicitly changed to `1` there. The statute file's `save_terms => 0` appears to be a leftover from before that convention was established, never updated in the 3.5.0 overhaul. The docblock for v3.5.0 does not mention changing `save_terms`.

**Recommendation**: Change all taxonomy fields in `acf-jx-statutes.php` from `save_terms => 0` to `save_terms => 1` (and `load_terms => 1` is already set). Re-run a migration pass to populate term relationships for existing statute posts.

## 🟡 Inconsistencies / Design Concerns

### 4. `acf-source-verify.php` — `ws-legal-update` missing from location rules

The `acf-source-verify.php` header says attached CPTs are: `jx-statute, jx-citation, jx-interpretation, ws-agency, ws-assist-org, jx-summary, ws-reference`. But `ws-legal-update` is absent. Legal update records have a `source_method` concept too — they're created either by humans, the major-edit hook, or potentially by ingest tooling. If ingest ever creates `ws-legal-update` posts programmatically, there's no source provenance tracking on them.

This may be intentional (legal updates are always internally generated, not ingested), but it's worth a conscious decision documented in the header. **The exclusion is not currently documented** in the EXCLUDED CPTs note, which only lists the omission of `ws-legal-update` from `acf-plain-english-fields.php` in that file. `acf-source-verify.php` has no such exclusion rationale for `ws-legal-update`.

### 5. `acf-jx-interpretations.php` — `ws_jx_interp_affected_jx` with `save_terms => 0` and `load_terms => 0`

The affected jurisdictions field is intentionally not written to the taxonomy table (`save_terms => 0`) — it stores term IDs in post meta. The auto-population hook `ws_interp_auto_populate_affected_jx()` writes to `ws_jx_interp_affected_jx` meta directly. This is coherent *for the hook*, but the ACF field UI will never show a pre-selected value in the multi-select because `load_terms => 0` and the field's stored meta format (a serialized array of IDs) may not match what ACF expects when rendering the selector. Editors who need to manually override the affected jurisdictions may see a blank field even when the meta is populated.

**Recommendation**: If manual override is a real use case, either set `load_terms => 1` and let the hook write to the taxonomy table too, or switch to `save_terms => 0 / load_terms => 0` but use `acf/load_value` to pull from post meta directly (the same pattern used for `ws_citation_prefill_statute_ids`).

### 6. `acf-references.php` — filename/docblock mismatch

The filename is `acf-references.php` but the docblock title reads `acf-ws-reference.php`. Cosmetic, but worth a one-line fix.

### 7. `acf-jx-statutes.php` — `ws_jx_statute_last_reviewed` is outside all tabs

`field_jx_statute_last_reviewed` is declared after the Reference Materials tab block, meaning it renders outside the tab structure — it will appear loose at the bottom of the metabox rather than inside any tab. Every comparable "Last Verified Date" field in other CPT files (`acf-agencies.php`, `acf-assist-orgs.php`, `acf-jx-citations.php`, `acf-jx-interpretations.php`) has the same pattern of being content-owned and outside stamps, but in those files there's typically only one active tab group so "outside tabs" is fine. In `acf-jx-statutes.php` with 7 named tabs, a field floating after the last tab may be invisible to editors in practice.

**Recommendation**: Move `field_jx_statute_last_reviewed` inside the Links tab (or give it its own tab) so it isn't orphaned.

### 8. `acf-plain-english-fields.php` — `field_plain_english_reviewed` key collision with `acf-jx-summaries.php`

Both files register a field with `'key' => 'field_plain_english_reviewed'` and `'name' => 'ws_plain_english_reviewed'`. In `acf-plain-english-fields.php` this attaches to `jx-statute, jx-citation, jx-interpretation, ws-agency`. In `acf-jx-summaries.php` it attaches to `jx-summary` only. Since a `jx-summary` post will never load the plain-english group (it's excluded), there's no ACF collision at runtime. However, if ACF ever validates field keys globally (some versions do), this duplicate key could cause warnings or silently suppress one registration.

**Recommendation**: Rename the key in `acf-jx-summaries.php` to something like `field_jx_sum_plain_reviewed` to make the keys unique, even though the meta name stays `ws_plain_english_reviewed`.

Similarly, `field_plain_english_by` and `field_plain_english_reviewed_by` exist in both files with identical keys. Same risk, same fix.

### 🔴 Key collision with `acf-jurisdictions.php`

`acf-stamp-fields.php` registers `field_create_author` and `field_date_created` as ACF field keys. `acf-jurisdictions.php` also registers fields with those exact same keys in its Record Management tab (`field_create_author`, `field_date_created`, `field_last_edited_author`, `field_last_edited`). ACF field keys are supposed to be globally unique across all registered field groups. Since jurisdiction is excluded from the stamp group's location rules, there's no runtime UI collision — both groups won't appear on the same screen — but ACF may log warnings or behave unpredictably if it encounters duplicate keys during `acf_add_local_field_group` registration. This is the same category of risk as the duplicate `field_plain_english_reviewed` keys flagged in the prior review.

**Recommendation**: Namespace the jurisdiction-specific versions in `acf-jurisdictions.php` to `field_jx_create_author`, `field_jx_date_created`, `field_jx_last_edited`, `field_jx_last_edited_author`. The meta names (`ws_auto_*`) can stay identical — only the ACF field key needs to be unique.
