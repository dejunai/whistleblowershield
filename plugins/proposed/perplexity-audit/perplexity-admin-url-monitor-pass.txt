## `admin-url-monitor.php` — Findings

### ✅ Security Posture — Solid

Manual trigger is properly gated: `check_admin_referer()`, `current_user_can( 'manage_options' )`, and nonce verified before `ws_run_url_health_check()` runs . Dashboard widget registration checks capability before registering. All output in the widget is escaped.

***

### 🟡 Should Fix

**1. `ws_run_url_health_check()` — unbounded `get_posts( 'posts_per_page' => -1 )` across three CPTs**
Every cron run fetches all published posts for `jurisdiction`, `ws-agency`, and `ws-assist-org` with no limit . For a site with 57 jurisdictions and a few hundred agencies/orgs this is fine today. If assist-org volume grows significantly, this becomes a memory spike on a cron thread. Worth adding a comment noting the known ceiling, or batching by post type with a transient checkpoint — but not urgent at current scale.

**2. `ws_url_monitor_map` is a module-level global consumed via `global` inside `ws_run_url_health_check()`**
Same pattern as `$ws_stamp_cpts` and `$_ws_dir_*` . Consistent across the codebase now, noted for the record.

**3. `ws_url_monitor_deactivate()` is defined but never hooked**
The docblock says "Hook this to `register_deactivation_hook()` in your main plugin file" , but there's no evidence in the files reviewed so far that `ws-core.php` actually calls it. If it isn't hooked, the cron event persists after plugin deactivation and fires against a missing callback, producing PHP notices. Verify this is wired in `ws-core.php`.

***

### 🔵 Tech Debt

**4. Warning detection logs a fresh `detected` timestamp on every run**
When a redirect warning persists across multiple runs, the `detected` field gets overwritten with the current timestamp on each check  — because the log is rebuilt from scratch and the entry is re-added with `current_time()`. The original detection date is lost after one run. The failure path has the same issue. If you ever want "detected X days ago" in the widget, the `detected` value from `$previous_flagged` should be preserved on re-log.
