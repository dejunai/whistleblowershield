## Pass 1 — Audit: `query-helpers`, `query-shared`, `shortcodes-general`, `render-general`, `render-jurisdiction`, `render-section`

### 🔴 Critical

**1. `render-jurisdiction.php` — `ws_get_jx_statute_data()` called instead of `ws_get_jx_statutes()`**
The curated assembler calls `ws_get_jx_statute_data( $jx_term_id )` as the publish gate for the statutes section , but `ws_is_published()` itself documents that the statutes query function returns an **array-of-arrays** (state + federal merge) and the function is named `ws_get_jx_statutes()` (plural) . If `ws_get_jx_statute_data()` is a different singular function that returns only a single record, the gate is wrong — it would miss the federal fallback. If it doesn't exist at all, this is a silent fail returning `false` and the statutes section never renders. This needs to be verified against `query-jurisdiction.php` on the next pass — it's the highest priority cross-file check.

**2. `shortcodes-general.php` — `$_GET['post_id']` in `[ws_reference_page]` is not validated against allowed post types**
The shortcode accepts `post_id` from `$_GET` and passes it directly to `ws_get_reference_page_data()` . There's no check that the post ID belongs to `jx-statute`, `jx-citation`, or `jx-interpretation` before calling the data function. If someone crafts a URL with `?post_id=1` (the WP home page post), the query layer function would need to guard this — it may or may not. Until `query-jurisdiction.php` is read, this is a trust-the-query-layer assumption worth confirming.

***

### 🟡 Should Fix

**3. `render-section.php` — `ws_render_jx_citations()` emits raw `$item` HTML with no escaping**
Each `$item` in the loop is echoed directly: `echo $item;` . The comment says items are "pre-built HTML strings" from the shortcode, but there's no `wp_kses_post()` at the render layer boundary. The pattern elsewhere in this file is to apply `wp_kses_post()` on content passed from shortcodes (see `ws_render_section()`) — citations breaks that pattern. The fix is to either wrap in `wp_kses_post()` here or explicitly document in the @param that callers are responsible for escaping.

**4. `render-section.php` — `ws_render_jx_summary_section()` echoes `$review_html` with no escaping**
The `$review_html` footer block is echoed raw inside `<footer>` . It comes from `ws_render_jx_summary_footer()` which also lives in this file, so the trust boundary is narrow — but the pattern is inconsistent with the `wp_kses_post()` applied to `$content` just above it. Should be `echo wp_kses_post( $review_html )` or documented as trusted internal HTML.

**5. `shortcodes-general.php` — `date( 'Y' )` in `[ws_footer]` is not timezone-aware**
The footer shortcode passes `'year' => date( 'Y' )` to the render function . PHP's `date()` uses the server timezone, not WordPress's configured timezone. On a server set to UTC close to midnight on Dec 31, visitors in US timezones could briefly see the wrong year. The fix is `date_i18n( 'Y' )` or `wp_date( 'Y' )`, which both respect `get_option('timezone_string')`.

**6. `render-section.php` — `ws_render_plain_english_reviewed_badge()` uses `esc_attr()` instead of `esc_html()` for tooltip content injected into a `title` attribute**
The reviewer name and date are run through `esc_attr()` before being inserted into `title="..."` . This is technically correct for attribute context, but `esc_attr()` on user display names that contain HTML entities (e.g. `O'Brien`) will double-encode on output since the value has already been retrieved from the DB. `esc_html()` would be the appropriate escape here — it produces the same result for attribute context when the surrounding quotes are already in place.

**7. `render-jurisdiction.php` — `[ws_legal_updates]` passes `$post->post_name` (slug) as the `jx` attribute**
The shortcode handler in `shortcodes-general.php` accepts either a numeric post ID or a USPS code string and calls `ws_get_id_by_code( strtoupper( $atts['jx'] ) )` to resolve it . The assembler passes the post *slug* (e.g. `california`) not the USPS code (`CA`) . `ws_get_id_by_code()` looks up by taxonomy code, not by slug — so passing `california` will return `0`, the query will be sitewide, and `$jx_id = 0` means no jurisdiction scoping is applied. The legal updates section will show all recent updates instead of jurisdiction-scoped ones. The assembler should pass the jurisdiction's `ws_code` meta value, not `$post->post_name`.

### 🔵 Low Risk / Tech Debt

**8. `query-shared.php` — `ws_build_plain_english_array()` always reads `ws_has_plain_english` even for `jx-summary` posts where the field is not used**
The file comment acknowledges this: *"jx-summary calls this function but ignores has_plain_english"* . It works correctly — the unused key is simply ignored by the render layer — but it wastes one `get_post_meta()` call per jx-summary render. Minor, but worth noting if query performance becomes a concern.

**9. `render-general.php` — `ws_render_footer()` hardcodes `admin@whistleblowershield.org`**
The contact email is a literal string in the render function . Every other configurable string in the plugin uses `apply_filters()` or a constant. If the email ever changes, this needs a code edit. A `apply_filters( 'ws_footer_contact_email', 'admin@...' )` would make it overridable without a code change.

**10. `query-helpers.php` — only one function, no `is_int()` check before `(int)` cast**
`ws_resolve_display_name()` casts `$user_id` to `int` before the falsy check . Passing a non-numeric string like `'abc'` would cast to `0` and correctly return `''`. This is fine — just noting the reliance on cast behavior rather than explicit input validation, consistent with WordPress core conventions.

## Pass 2 — `shortcodes-jurisdiction.php` + `query-jurisdiction.php`

### 🔴 Critical — New

**1. `query-jurisdiction.php` — `WS_JURISDICTION_TERM_ID` constant used as taxonomy slug everywhere**
Every `tax_query`, every `wp_get_post_terms()`, and every `get_term_by()` call uses `WS_JURISDICTION_TERM_ID` as the taxonomy name . That constant is also used as the **term ID integer** in `ws_get_jx_term_id()` (`wp_get_post_terms( $post_id, WS_JURISDICTION_TERM_ID )`). The taxonomy is registered as `'ws_jurisdiction'` per `register-taxonomies.php`  and the constant's actual value needs to be verified from `ws-core.php`. If `WS_JURISDICTION_TERM_ID` resolves to an integer (like `4`), then every taxonomy query in the entire plugin is broken — `'taxonomy' => 4` is not a valid taxonomy slug. This is the single most consequential thing to verify before launch.

**2. `query-jurisdiction.php` — `ws_get_term_id_by_code()` also uses `WS_JURISDICTION_TERM_ID` as taxonomy slug**
Directly related to #1: `get_term_by( 'slug', $slug, WS_JURISDICTION_TERM_ID )` . If the constant is an integer, this call returns `false` for every code lookup, breaking `ws_get_id_by_code()`, which breaks jurisdiction scoping in `[ws_legal_updates]`, `[ws_jx_flag]`, and any external caller passing a code string.

### 🟡 Should Fix

**3. `query-jurisdiction.php` — `ws_get_jx_summary_data()` returns draft/pending posts**
The summary query uses `'post_status' => [ 'publish', 'draft', 'pending' ]`  — intentional for admin preview, but `ws_is_published()` in `render-jurisdiction.php` gates on `status === 'publish'` . That gate works correctly, but if any other caller uses `ws_get_jx_summary_data()` without checking `status`, a draft summary could leak into output. This is a trust-the-callers pattern that's fragile as the codebase grows.

**4. `shortcodes-jurisdiction.php` — `[ws_jx_header]` silently renders empty gov links**
The header shortcode builds `$gov_data['links']` from `portal_url`, `executive_url`, and `authority_url` . `ws_render_jx_gov_offices()` skips items where `$link['url']` is empty  — good. But `$box_label` falls through to `'Official Leadership Offices'` for any `class` value not in the `$labels` array . Since `class` comes from `get_field( 'ws_jurisdiction_class' )`, a misconfigured or null field silently renders a generic label. Low impact but worth logging at `WP_DEBUG` level.

**5. `query-jurisdiction.php` — `ws_get_nationwide_assist_org_data()` has no `tax_query` relation key**
When `$filters['type']`, `sector`, and `stage` are all set, multiple `tax_query` entries are appended without a top-level `'relation' => 'AND'` key . Without it, WP defaults to `AND` — so it works correctly today — but the missing explicit declaration is a latent bug if WP's default ever changes or if the code is copied to a context where the relation matters. The `meta_query` correctly has `'relation' => 'AND'` . The `tax_query` should match.

**6. `query-jurisdiction.php` — `ws_get_legal_updates_data()` sitewide cache ignores `$count`**
The function resolves `$count` before checking the transient, but the sitewide transient stores results without encoding `$count` in the key . If the first call is `ws_get_legal_updates_data( 0, 10 )`, 10 items are cached under `WS_CACHE_LEGAL_UPDATES_SITEWIDE`. A subsequent `ws_get_legal_updates_data( 0, 100 )` returns only 10. Currently the only sitewide caller uses the default `$count = 0` (resolves to 100), so this is dormant — but a second sitewide caller with a different count would get a stale/wrong result.

**7. `query-jurisdiction.php` — `ws_get_jx_citation_data()` `label` fallback chain reads `ws_jx_citation_common_name` twice**
```php
'label' => get_post_meta( $cid, 'ws_jx_citation_common_name', true )
        ?: get_post_meta( $cid, 'ws_jx_citation_official_name', true )

```
The first key is `ws_jx_citation_common_name` and the intended fallback should be `official_name`, then title . But `ws_jx_citation_official_name` is actually the correct second fallback — so the chain works. However, `common_name` is also independently read two lines above: `'common_name' => get_post_meta( $cid, 'ws_jx_citation_common_name', true )`. That's two `get_post_meta()` calls for the same key per citation. Minor performance waste; deduplicate by reading into a local variable.

### 🟡 Should Fix

**1. `ws_render_directory_card()` — `$org['services']` is always a string, never an array**
The card checks `is_array( $org['services'] )` and iterates it as tag pills . But `ws_get_assist_org_data()` and `ws_get_nationwide_assist_org_data()` both read `services` via `get_post_meta( $oid, 'ws_aorg_services', true )`  — `get_post_meta` with `$single = true` returns a string, not an array. So `is_array()` is always false, the services block never renders, and no error is thrown. The fix depends on how `ws_aorg_services` is stored: if it's a serialized array (ACF checkbox/multi-select), use `get_field()` instead; if it's a comma-separated string, split it here.

**2. `ws_render_directory_card()` — `$_ws_dir_cost_labels` and `$_ws_dir_service_labels` are module-level globals**
The card function pulls them in via `global` . This works fine solo, but both arrays are defined at file scope without being wrapped in a function or constant — they're effectively anonymous globals with `$_` prefixes as the only signal. If any other file ever defines a variable with the same name before this file loads, values get stomped silently. Low risk given load order, but worth converting to constants or a static accessor when convenient.
