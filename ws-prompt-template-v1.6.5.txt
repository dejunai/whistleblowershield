═══════════════════════════════════════════════════════════════════════════════
SECTION 1 — CRITICAL OPERATING RULES
═══════════════════════════════════════════════════════════════════════════════

OMISSION vs. EMPTY
A missing key is a functional instruction to the ingest tool to preserve
existing database values. Apply this distinction strictly:

  OMIT THE KEY ENTIRELY if research is inconclusive or the data is unconfirmed.
  USE "" or [] only if you have confirmed the value is explicitly empty,
    non-existent, or "None" (e.g. a statute with no reward program, no tolling).
  NEVER use null, "N/A", or any placeholder string.

NO REDUNDANCY
Do not repeat data across fields. Specifically:
  adverse_action_scope  → defines the legal threshold (standard of harm)
  adverse_action array  → taxonomy slugs of specific acts only
These two fields describe different things. Do not duplicate content between them.

FIELD ORDER
Maintain strict field order within meta and within each record exactly as shown
in the schemas. Do not reorder keys. Do not add keys not present in the schema.

CALCULATED FIELDS
Calculate all count keys (record_count, proposed_count, citation_count,
error_count) only after their respective arrays are fully finalised.

OUTPUT CONSTRAINTS
json_run_notes is the only field permitting editorial latitude.
All other fields are strictly schema-constrained.
Output only the JSON object inside a single code block.
No commentary, markdown prose, or explanation outside the code block.

SOURCE INTEGRITY
Prioritise .gov and official legislative sources for statute_url.
For citations, prioritise appellate and supreme court decisions.
Omission of citations for established, litigated statutes is a quality failure —
include at least one high-authority case per record where case law exists.

SECTION 2 — TAXONOMY RULES
═══════════════════════════════════════════════════════════════════════════════

2.1 ABSOLUTE CONTROLLED VOCABULARY
All taxonomy array values must match the predefined slugs in Section 4 exactly
(kebab-case, lowercase). Do not modify, pluralise, or abbreviate slugs.
If a needed value is not in Section 4, it must not enter the record —
move it to new_terms_proposed in the meta block.

2.2 COMBINATORIAL CLASSIFICATION (Anti-Fracture Rule)
Before proposing a new term, determine whether the legal concept can be
accurately represented by combining existing terms.
Example: A statute protecting healthcare workers reporting fraud should use
  clinical-staff + healthcare-medicare-fraud
Do not propose: healthcare-fraud-whistleblower
Propose a new term ONLY if the concept is a distinct legal species that
cannot be described by a combination of existing traits.

2.3 MULTI-TAGGING REQUIREMENT
Assign all applicable tags. A statute is rarely a single-issue law.
Include the subject matter, protection type, and specific procedure where all apply.

2.4 PRECISION REQUIREMENT
Close enough is not acceptable for legal data. If an existing term is so
broad it loses legal meaning for a specific statutory class, a new term
must be proposed.

2.5 WHEN TO PROPOSE A NEW TERM
Propose a term only if:
  - A recurring, legally meaningful class is genuinely missing
  - The existing vocabulary requires more than 4 tags to describe one concept
  - A specific statutory definition creates a unique class with distinct
    SOL or remedy rules that differ materially from existing terms

2.6 WHEN NOT TO PROPOSE A NEW TERM
Do NOT propose terms for:
  - Synonyms of existing terms
  - Subsets of a category already covered by a different taxonomy
  - One-off examples unlikely to recur across other jurisdictions

2.7 PROPOSAL INTEGRITY
Every entry in new_terms_proposed must:
  - Include both term_label (human-readable) and term_id (kebab-case)
  - Provide a notes justification explaining why existing tags could not
    be combined to represent the concept
  - List all statutes in the current batch where this term was seen_in
  - Do not propose new taxonomy tables — recommend in json_run_notes instead
  - Do not propose new taxonomy parents — recommend in json_run_notes instead

BURDEN_OF_PROOF NOTES:
  burden_of_proof_details should be used whenever the burden framework involves
  multi-step analysis, burden shifting, statutory interpretation, or significant
  case law clarification. For most whistleblower statutes this field is expected
  to be non-empty.
  burden_of_proof_flag = true only when:
    1. The statute explicitly defines burden shifting, OR
    2. Case law materially alters the default legal standard

REWARD NOTES:
  available = true only where the statute explicitly provides a financial award,
  bounty, or percentage-based recovery to the reporting individual.
  Anti-retaliation remedies (back pay, damages, reinstatement) do NOT qualify.
  Qui tam provisions, bounty programs, and statutory award schemes qualify.

CITATION NOTES:
  Prioritise appellate and supreme court decisions over trial-level cases.
  Prefer cases that directly interpret or apply the cited statute.
  Each entry must follow this exact format:
    "Case Name v. Case Name || SPECIFIC_IMPACT || URL || Source || QUALITY"

  SPECIFIC_IMPACT: a short precise phrase (3-8 words) describing functional
  legal impact. Reference one of: disclosure, retaliation, burden, causation,
  protected activity, adverse action. Use one of these patterns:
    "defines [legal concept]"         "clarifies [legal standard]"
    "establishes [rule/test]"         "applies [statute/standard]"
    "limits [scope/protection]"       "expands [scope/protection]"
    "interprets [term/phrase]"        "confirms [legal principle]"
    "rejects [legal argument]"        "resolves [conflict/ambiguity]"
  Avoid vague phrases: "relevant case", "important decision", "applies law"

  QUALITY values:
    high     = appellate/supreme court, frequently cited
    moderate = appellate but narrower or less cited
    low      = trial-level or limited precedential value

  URL must link to a publicly accessible source (court website, Google Scholar,
  Justia, CourtListener, or equivalent).
  Do not duplicate citations across records unless directly relevant to multiple statutes.
  If no reliable citation can be confirmed: use an empty array [].
