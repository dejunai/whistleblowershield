AUDIT CATEGORIES
For every field in every file, evaluate all four categories and report findings. If a field has no issues in a category, note it as clean — do not skip it entirely. A complete report means every field is accounted for.
Category 1 — Field Attributes
Check for:

Redundant attributes — attributes explicitly set to their ACF default value and therefore unnecessary
Missing attributes that would improve data integrity or editorial experience — examples include min/max on number fields, max_rows on repeaters, return_format on date and user fields, allow_null on required relationship fields, multiple on taxonomy fields where multiple selection is logical
Inconsistent attributes across similar fields — if ws_jx_statute_date_created uses one date format and ws_jx_cite_date_created uses a different one, flag it
Missing wrapper or placeholder attributes where they would meaningfully aid the editorial experience

Category 2 — Field Types
Check for:

Fields using a suboptimal type for the data being stored — examples: a field storing a URL typed as text instead of url, a field storing a yes/no value typed as text instead of true_false, a field storing a numeric value typed as text instead of number, a field storing structured repeating data as a textarea instead of a repeater
Fields where the current type technically works but a more specific type would improve validation, editorial experience, or frontend output
Fields where the name is technically compliant but inconsistent. example: Fieldname uses _website instead of _url
Date fields — verify all date fields use a consistent format and appropriate type
Relationship fields — verify post_type filters are correctly scoped and return_format is explicitly set
Taxonomy fields — verify taxonomy, field_type, and return_format are explicitly set on every taxonomy field
Select fields — recommend change to 'taxonomy' if there is a logical benfit
User fields — verify return_format is explicitly set

Category 3 — Prefix Review
Evaluate every occurrence of ws_ and jx_ prefixes across field names, field keys, and group keys:

Field name — this becomes the wp_postmeta meta_key. Namespacing prefixes serve a genuine collision-prevention purpose here. Flag only if the prefix is genuinely redundant given the field's scope — for example if a field is already unambiguous within its CPT context and collision risk is negligible.
Field key — ACF internal identifier. ws_ prefix here is purely noise and adds no collision protection within ACF's own namespace. Flag every field key carrying a ws_ prefix as a candidate for removal.
Group key — same as field key. Flag every group key carrying a ws_ or jx_ prefix as a candidate for removal.
jx_ prefix specifically — evaluate separately from ws_. Where jx_ carries semantic meaning about the record's jurisdiction-association, note it as semantically valuable even if technically redundant. Where it is purely noise, flag it.

Category 4 — Instructions
For every field, evaluate whether instructions are present and adequate:

Missing instructions — flag any field where the purpose, expected format, or relationship to other fields is not immediately obvious to a non-technical contributor
Unclear instructions — flag any existing instruction that is ambiguous, overly technical, or assumes knowledge a contributor might not have
Unnecessary instructions — flag any instruction on a field so self-explanatory that the instruction adds noise rather than clarity
Pay special attention to fields involving legal concepts — ws_jx_statute_trigger, ws_jx_statute_exhaustion_required, ws_statute_burden_of_proof, ws_jx_statute_tolling_notes and similar — these are the highest priority for clear contributor guidance
