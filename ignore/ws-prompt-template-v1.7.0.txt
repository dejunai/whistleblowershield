SECTION 1 — OUTPUT FORMAT & OMISSION LOGIC
═══════════════════════════════════════════════════════════════════════════════

THE OMIT RULE
If research is inconclusive or a data point cannot be confirmed: OMIT THE KEY
ENTIRELY. A missing key is a functional instruction to the ingest tool to
preserve existing human-verified data in the database.
  DO NOT use "" or [] to mean "I don't know."
  DO NOT use null, "N/A", or any placeholder string.

THE EMPTY RULE
Use "" or [] ONLY if you have confirmed the value is explicitly empty,
non-existent, or legally none (e.g. a statute that explicitly provides
no reward program, no tolling provisions).

NO REDUNDANCY
Do not repeat data across fields. Specifically:
  adverse_action_scope → the legal threshold or standard of harm (string)
  adverse_action array → taxonomy slugs of specific acts only
These describe different things. Do not duplicate content between them.

FIELD ORDER
Maintain strict field order within meta and within each record exactly as
shown in the schemas. Do not reorder keys. Do not add keys not present
in the schema.

CALCULATED FIELDS
Calculate all count keys (record_count, proposed_count, citation_count,
error_count) only after their respective arrays are fully finalised.
record_count must reflect the actual number of completed records in the
array — not the requested count. If asked for 15 and 12 were completed,
record_count must be 12.

OUTPUT FORMAT
Output only the JSON object inside a single code block. No commentary,
markdown prose, or explanation outside the code block.

SOURCE INTEGRITY
Prioritise .gov, .ca.gov, and official legislative domains for statute_url.
For citations, prioritise appellate and supreme court decisions.
Omission of citations for established, litigated statutes is a quality
failure — include at least one high-authority case per record where
case law exists. If no reliable citation can be confirmed, use [].

SECTION 2 — TAXONOMY RULES (WHITELIST LOCKDOWN)
═══════════════════════════════════════════════════════════════════════════════

2.1 NO UNAUTHORIZED SLUGS

You are FORBIDDEN from placing invented, adapted, pluralised, or approximated
slugs directly in any record array. Every taxonomy value in every record must
be an EXACT verbatim match to a slug listed in Section 4.

If a concept is needed but no matching slug exists in Section 4:
  1. OMIT the value from the record array entirely — do not substitute
  2. PROPOSE the term in meta -> new_terms_proposed
  DO NOT use a parent slug as a placeholder for a missing child slug.
  DO NOT use human-readable labels (e.g. "Back Pay") — use back-pay.

There are no exceptions. Inventing a slug and placing it in a record
corrupts the database. Propose it instead.

2.2 LEAF-NODE PREFERENCE (No Parent Slugs in Records)

Parent slugs are structural labels only. They organise the taxonomy visually.
They must NEVER appear in any record array.

⚠ EXAMPLE OF VIOLATION:
  "protected_class": [ "public-sector" ]        ← FORBIDDEN — parent slug

⚠ CORRECT PATTERN:
  "protected_class": [ "state-employee", "local-gov-staff" ]  ← child slugs

Always drill down to the most specific child slug available. If no child
slug is specific enough, propose a new term via new_terms_proposed.

2.3 COMBINATORIAL CLASSIFICATION (Anti-Fracture Rule)

Before proposing a new term, determine whether the legal concept can be
accurately represented by combining 3 or fewer existing slugs across the
relevant taxonomy fields.

  EXAMPLE: A statute protecting healthcare workers reporting fraud should use:
    clinical-staff + healthcare-medicare-fraud  ← correct combination
  DO NOT propose: healthcare-fraud-whistleblower  ← invented species

Propose a new term ONLY if:
  - The concept is a distinct legal species not describable by 3 or fewer traits
  - A recurring, legally meaningful class is genuinely absent
  - A specific statutory definition creates a class with materially different
    SOL or remedy rules from any existing term
  - The existing vocabulary requires more than 4 slugs to describe one concept

DO NOT propose terms for:
  - Synonyms of existing terms
  - Subsets of a category already covered by a different taxonomy
  - One-off examples unlikely to recur across other jurisdictions

2.4 MULTI-TAGGING REQUIREMENT

Assign all applicable slugs. A statute is rarely a single-issue law. Include
the subject matter, protection type, and specific procedure where all apply.

2.5 PROPOSAL INTEGRITY

Every entry in new_terms_proposed must:
  - Include both term_label (human-readable) and term_id (kebab-case)
  - Explain in notes why existing slugs cannot be combined to represent it
  - List all statutes in the current batch in seen_in
  - Do not propose new taxonomy tables — recommend in json_run_notes
  - Do not propose new taxonomy parents — recommend in json_run_notes

CITATION FORMAT:
  "Case Name v. Case Name || SPECIFIC_IMPACT || URL || Source || QUALITY"

  protected activity, adverse action. Use one of these patterns:
    "defines [legal concept]"         "clarifies [legal standard]"
    "establishes [rule/test]"         "applies [statute/standard]"
    "limits [scope/protection]"       "expands [scope/protection]"
    "interprets [term/phrase]"        "confirms [legal principle]"
    "rejects [legal argument]"        "resolves [conflict/ambiguity]"

  ✓ CORRECT:   "confirms exhaustion requirement"
  ✓ CORRECT:   "defines adverse employment action"
  ✗ INCORRECT: "Exhaustion of administrative remedies"  ← noun phrase, no verb
  ✗ INCORRECT: "Anti-immigration retaliation scope"     ← noun phrase, no verb

  QUALITY values:
    high     = appellate/supreme court, frequently cited
    moderate = appellate but narrower or less cited
    low      = trial-level or limited precedential value

  approximate URLs. If a reliable URL cannot be confirmed, omit the URL
  and note the citation as unverified in json_run_notes. An unverified
  citation noted in json_run_notes is preferable to a false URL.

  Prioritise appellate and supreme court decisions over trial-level cases.
  Do not duplicate citations across records unless directly relevant to both.
